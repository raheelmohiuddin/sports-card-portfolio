## Methodology that worked

**Ground-truth-first** — before any `cdk deploy` or destructive action, verify current AWS state directly via CloudTrail, drift detection, `get-template`, `describe-*`. Don't operate on session-memory of what the state "should be." This pattern surfaced the 2026-05-21/22 Phase 4b wrong-writer binding via drift detection rather than narrative recall.

**Diff review as gate** — every `cdk diff` output reviewed line by line before deploy. Bundled-scope diffs surfaced at this gate: Path A's em-dash + alarm-dimension side effects (accepted explicitly), the IDOR fix's clean 2-Lambda diff (no surprises), the CORS+em-dash bundle's exact match to expected scope (`cdk diff --strict` to expose CDK's default suppression of "mangled non-ASCII" diffs).

**Verify deploy by reading stored state** — `UPDATE_COMPLETE` means the underlying AWS API call succeeded, not that the requested string persisted as bytes. `get-template` (or equivalent state read) is the post-deploy verification. This caught the em-dash mojibake on first post-Path-A check, even though Path A's deploy log reported `UPDATE_COMPLETE` on every affected resource.

**AWS-doc verification before mechanism claims** — when investigating CFN behavior (DeletionPolicy: Retain semantics, SecretTargetAttachment deletion handler, em-dash conversion layer), pull the authoritative docs and trace actual bytes rather than relying on plausible-sounding explanations. The earlier-session GenerateSecretString-regenerates-on-detach and `--import-existing-resources`-matches-by-prefix mechanism claims were both contradicted by AWS docs once verified.
