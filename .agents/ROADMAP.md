- **Database indexing — 3 unindexed FK columns at scale** — Audit 2026-05-23 identified three foreign-key columns lacking efficient indexes:

  (1) `cards.sold_show_id` — FK to card_shows; no index. Used by `ON DELETE SET NULL` cascade when a card_shows row is deleted (rare but possible at scale).

  (2) `consignments.card_id` — FK to cards; no index. Used by `ON DELETE CASCADE` when a card is deleted. Sequential scan of consignments on each card deletion.

  (3) `user_shows.card_show_id` — FK to card_shows. The existing UNIQUE index `(user_id, card_show_id)` doesn't serve `WHERE card_show_id = X` queries because card_show_id isn't the leftmost column. Affects `ON DELETE CASCADE` when shows are retired.

  At current scale (14 cards, 2 consignments, 17 user_shows), impact is negligible — sequential scans complete in microseconds. At scale (1K+ rows per table), CASCADE deletes start to degrade.

  Fixes:
  ```sql
  CREATE INDEX idx_cards_sold_show_id ON cards(sold_show_id) WHERE sold_show_id IS NOT NULL;
  CREATE INDEX idx_consignments_card_id ON consignments(card_id);
  CREATE INDEX idx_user_shows_card_show_id ON user_shows(card_show_id);
  ```

  The audit also showed several indexes with 0 scans in the past ~3 days. Some are domain-correct ("hasn't been exercised yet" — e.g., `idx_cards_sold_at` only matters when cards are sold), but worth re-evaluating which are truly dead after 3+ months of real traffic. Index inventory and usage stats live in `pg_indexes` and `pg_stat_user_indexes` respectively.

  Methodology: full index inventory + table row counts + index-usage statistics + FK constraint listing via Aurora Data API.

  Trigger: post-first-non-Raheel-user OR post-1K-rows-per-table OR post-10-user-base (whichever earliest). Size: S (~30 min: write 3-statement migration, deploy via existing migration tooling, verify via EXPLAIN ANALYZE on affected query patterns).
