Aurora supports `StorageEncrypted` only at cluster creation; flipping requires snapshot → restore-to-new-encrypted-cluster → cutover. Recovery procedures change once encryption ships because restored snapshots inherit encryption from the source — PITR-restore commands will need additional KMS flags (see [OPERATIONS.md §1 Known gaps](./OPERATIONS.md)). - **Git-vs-deployed Lambda source drift detection** — no signal currently exists for "backend Lambda source has commits past the deployed asset hash." Manual `cdk deploy` + auto-Amplify-on-push can desync indefinitely. Drift was discovered manually during P0 Hardening Session A's cdk-diff gate (2026-05-14) — ~13-hour Sev-3 contract drift between frontend (sending `sellTargetPrice`) and 5 backend Lambdas (still on pre-`9cbf9cc` code returning `targetPrice`); see [.agents/p0-hardening-session-a-plan.md §3.5](./p0-hardening-session-a-plan.md). Without that gate, the drift was indefinitely-undetectable. Trigger: earliest of (a) CI/CD pipeline build (drift detection becomes a pipeline check), (b) first non-Raheel user (manual checks insufficient). Size: M (~3–4 hours; involves comparing git log per Lambda directory against deployed S3Key asset hashes, scriptable but needs careful design to avoid false positives during active development).
- **Frontend API call patterns + caching audit** — No systematic measurement of per-session API request count, per-page latency, or current caching state (the frontend uses raw `fetch` via `frontend/src/services/api.js`; no React Query / SWR / explicit client-side cache layer). Two concerns: (1) **Performance** — page-refresh and navigation latency on Portfolio / Card detail / Trade builder; whether serial request chains exist that could parallelize; perceived load time. (2) **Cost at scale** — current single-operator absolute cost is trivial, but session patterns scale linearly with users; redundant calls (multiple components fetching the same card, no request deduplication, no stale-while-revalidate) become expensive at 100+ users. Audit shape: DevTools network tab during canonical flows (portfolio load, refresh, modal open, navigation between pages, return to portfolio), count requests + identify duplicates + identify serial chains that could parallelize, map each request to its Lambda + DB query cost, classify per-data-type staleness tolerance (card prices: minutes OK; portfolio total: tighter; admin views: case-by-case). Techniques to evaluate post-audit: server-side `Cache-Control` headers, client-side query cache (React Query or SWR with stale-while-revalidate), in-flight request deduplication, lazy-loading + pagination for large card lists, conditional requests (ETag / If-Modified-Since), batching where multiple single-item fetches could be one. Constraint: **no UX regression from stale data** — per-data-type staleness tolerance must be locked before any caching ships; the audit produces the staleness matrix. Trigger: earliest of (a) monthly Lambda + API Gateway + RDS Data API spend crosses an investigate-worthy threshold (figure TBD at first month-end review post-second-user), (b) operator-perceptible or user-reported page-load latency, (c) first non-Raheel user (they'll exercise patterns Raheel doesn't), (d) ~100-user mark (tighter than CardHedger's ~500+ trigger because this is our own AWS spend, not upstream API spend). Size: M (investigation + plan + execution across frontend + possibly backend); potentially L depending on what the audit surfaces. Related: **CardHedger cost optimization** entry in Business / brand — both are cost-at-scale concerns but different surfaces (that's upstream API spend; this is our own AWS spend).
