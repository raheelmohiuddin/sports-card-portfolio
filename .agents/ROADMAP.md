- **`sport` column deprecation** — Keeping both `sport` and `category` columns indefinitely; revisit if migration friction emerges.
- **Amplify-managed Lambdas still on Node 20** — 4 Lambdas auto-generated by Amplify when auth was wired up (`amplify-login-custom-message-2e62d4e8`, `amplify-login-define-auth-challenge-2e62d4e8`, `amplify-login-create-auth-challenge-2e62d4e8`, `amplify-login-verify-auth-challenge-2e62d4e8`) remain on `nodejs20.x` after the Node 22 LTS upgrade because they're outside our CDK stack. Discovered post-deploy during Commit 1's nodejs20.x sweep (2026-05-19). Need to investigate Amplify CLI / Amplify Hosting Auth resource config to upgrade them. Binding deadline: 2027-03-03 (same as the rest, per Node 22 LTS entry now in Completed audits). Size: S (~1–2 hours including investigation).
- **Node 24 LTS upgrade — future consideration** — Pivot from Node 22 → Node 24 was evaluated 2026-05-19 immediately after the Node 22 upgrade landed. Recon recommended stopping at Node 22 tonight for three reasons: (1) Node 24.15.0 shipped 2026-04-15 with HTTP/stream/zlib use-after-free fixes — the line is still stabilizing; (2) five EOL'd APIs at v24.0 (`_headers`, `_headerNames`, `fs.truncate(fd)`, `tls.SecurePair`, `dirent.path`) — our application code is clean but transitive deps weren't grepped; (3) Lambda-specific community sentiment on nodejs24.x couldn't be verified during recon. Real upside: +12 months runway (Phase 3 deadline shifts from 2027-07-01 to 2028-07-01). Revisit triggers: (a) Node 22 runs 2+ weeks in production with no new incidents, (b) Node 24.x has another patch release, (c) transitive dep audit confirms zero v24-EOL'd API usage. Binding deadline if not upgraded: Node 22 Phase 3 deadline 2027-07-01. Size: S (~3 hours, same shape as the Node 20 → 22 upgrade per `.agents/node22-lts-upgrade-plan.md`).
- **RDS storage encryption migration** — cluster currently `StorageEncrypted: false`. Aurora supports `StorageEncrypted` only at cluster creation; flipping requires snapshot → restore-to-new-encrypted-cluster → cutover. Recovery procedures change once encryption ships because restored snapshots inherit encryption from the source — PITR-restore commands will need additional KMS flags (see [OPERATIONS.md §1 Known gaps](./OPERATIONS.md)). Trigger: earliest of (a) any PII/PHI scope expansion, (b) compliance requirement, (c) before first non-Raheel user. Size: M (multi-hour, downtime window required). Locked in [.agents/p0-hardening-session-a-plan.md §8 OQ-9](./p0-hardening-session-a-plan.md).
- **Custom RDS parameter group** — cluster currently on `default.aurora-postgresql16`. No immediate trigger; revisit when a parameter change is needed (logging tuning, slow-query log, etc.). Size: S. Locked in [.agents/p0-hardening-session-a-plan.md §8 OQ-9](./p0-hardening-session-a-plan.md).
- **Git-vs-deployed Lambda source drift detection** — no signal currently exists for "backend Lambda source has commits past the deployed asset hash." Manual `cdk deploy` + auto-Amplify-on-push can desync indefinitely. Drift was discovered manually during P0 Hardening Session A's cdk-diff gate (2026-05-14) — ~13-hour Sev-3 contract drift between frontend (sending `sellTargetPrice`) and 5 backend Lambdas (still on pre-`9cbf9cc` code returning `targetPrice`); see [.agents/p0-hardening-session-a-plan.md §3.5](./p0-hardening-session-a-plan.md). Without that gate, the drift was indefinitely-undetectable. Trigger: earliest of (a) CI/CD pipeline build (drift detection becomes a pipeline check), (b) first non-Raheel user (manual checks insufficient). Size: M (~3–4 hours; involves comparing git log per Lambda directory against deployed S3Key asset hashes, scriptable but needs careful design to avoid false positives during active development).
- **`application_name` not set on PG connection pool** — `backend/functions/_db.js`'s `getPool()` does not pass `application_name` in the connection config, so `pg_stat_activity` rows can't identify the calling Lambda. Forces NAT-gateway-IP + timing correlation during `RDS-Connections-High` triage (see [INCIDENT_RESPONSE.md §1](./INCIDENT_RESPONSE.md)). Fix: pass `application_name: process.env.AWS_LAMBDA_FUNCTION_NAME` in the Pool config. Trigger: earliest of (a) first `RDS-Connections-High` incident, (b) first non-Raheel user. Size: XS (~30 min — single-file change in `_db.js` + verification via Data API query against `pg_stat_activity`).
