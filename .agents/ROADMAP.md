(2) **Cost at scale** — current single-operator absolute cost is trivial, but session patterns scale linearly with users; redundant calls (multiple components fetching the same card, no request deduplication, no stale-while-revalidate) become expensive at 100+ users. Audit shape: DevTools network tab during canonical flows (portfolio load, refresh, modal open, navigation between pages, return to portfolio), count requests + identify duplicates + identify serial chains that could parallelize, map each request to its Lambda + DB query cost, classify per-data-type staleness tolerance (card prices: minutes OK; portfolio total: tighter; admin views: case-by-case). Techniques to evaluate post-audit: server-side `Cache-Control` headers, client-side query cache (React Query or SWR with stale-while-revalidate), in-flight request deduplication, lazy-loading + pagination for large card lists, conditional requests (ETag / If-Modified-Since), batching where multiple single-item fetches could be one. Constraint: **no UX regression from stale data** — per-data-type staleness tolerance must be locked before any caching ships; the audit produces the staleness matrix. Trigger: earliest of (a) monthly Lambda + API Gateway + RDS Data API spend crosses an investigate-worthy threshold (figure TBD at first month-end review post-second-user), (b) operator-perceptible or user-reported page-load latency, (c) first non-Raheel user (they'll exercise patterns Raheel doesn't), (d) ~100-user mark (tighter than CardHedger's ~500+ trigger because this is our own AWS spend, not upstream API spend).
